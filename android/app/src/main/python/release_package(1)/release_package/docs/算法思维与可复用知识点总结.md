## 品味记忆功能的可复用思想

**来源**: `PREFERENCE_FUNCTION_DEEP_DIVE.md`
**日期**: 2026-02-27

品味记忆功能的设计体现了几个重要的、可在不同项目中复用的算法思维和软件设计模式。

### 1. 用户反馈闭环 (User Feedback Loop)

这是将用户反馈（无论是显式的还是隐式的）整合进算法模型，形成一个持续优化的闭环系统。在本项目中，用户的“手动调节”就是一个强烈的反馈信号，表明了他们对“舒适”的定义。

-   **核心思想**: 算法不应是静态的、一成不变的。它应该能够“学习”用户的行为和偏好，并动态调整自身的参数或逻辑。
-   **可复用场景**:
    -   **推荐系统**: 根据用户的点击、收藏、跳过等行为，调整推荐权重。
    -   **智能家居**: 学习用户在不同时间段对灯光、温度的调节习惯。
    -   **UI/UX设计**: A/B测试后，根据用户在不同界面布局下的转化率，自动优化界面。

### 2. 关注相对量而非绝对量

算法的核心是记录和应用“压力比例”，这是一个相对量。在处理传感器数据时，相对量往往比绝对量更具优势。

-   **核心思想**: 绝对量容易受到传感器个体差异、老化、温度漂移、环境噪声等多种因素的干扰。而相对量（如比例、比率、变化率、梯度）在很大程度上抵消了这些共同模式的干扰，能更稳定地反映系统或对象的真实状态。
-   **可复用场景**:
    -   **图像处理**: 在光照不均的图像中进行目标检测时，使用梯度直方图（HOG）比使用像素绝对亮度值更鲁棒。
    -   **信号处理**: 在心率监测（PPG）中，通过分析信号的峰谷变化（AC/DC分量比）来计算血氧饱和度，而不是依赖信号的绝对强度。
    -   **金融风控**: 判断一笔交易是否异常时，考虑的是交易金额相对于该用户近期平均交易额的倍数，而不是交易金额的绝对值。

### 3. 默认逻辑与策略覆盖 (Default Logic & Strategy Override)

系统提供了一套普适的、可靠的默认调节逻辑，同时设计了清晰的接口来允许一个独立的“策略模块”(`PreferenceManager`)在特定条件下覆盖这个默认逻辑。这是一种经典的**策略设计模式 (Strategy Pattern)**。

-   **核心思想**: 将算法或业务逻辑的不同变体封装成独立的“策略”对象，使得它们可以互相替换。主系统持有一个对策略接口的引用，在运行时动态决定使用哪个具体策略。
-   **可复用场景**:
    -   **电商促销**: 计算订单价格时，可以根据用户等级、活动类型、优惠券等动态切换不同的“折扣策略”（如满减、打折、赠品）。
    -   **游戏AI**: NPC的行为可以根据其“性格”或“当前状态”（如巡逻、警惕、战斗）切换不同的“行为策略”。
    -   **数据存储**: 一个应用可以根据网络状况或数据类型，动态选择“本地存储策略”或“云端存储策略”。

### 4. 状态与逻辑分离

`PreferenceManager` 作为一个高内聚的模块，只负责管理“品味”的状态（记录了什么、当前是什么状态）和生成调节区间的逻辑。它不关心这些区间如何被使用。而 `IntegratedSeatSystem` 则负责使用这些区间来执行具体的气囊控制。这种分离使得两部分的职责都非常清晰。

-   **核心思想**: 一个模块应该只有一个引起它变化的原因。将“状态管理”和“业务逻辑执行”分离开，可以降低系统的耦合度，使各部分更易于独立测试、维护和复用。
-   **可复用场景**:
    -   **Web开发 (MVC/MVVM)**: Model层只负责管理数据和状态，View层负责展示，Controller/ViewModel负责协调。这是最经典的状态与逻辑分离。
    -   **机器人控制**: 一个模块负责路径规划（生成一系列路径点，即状态），另一个模块负责运动控制（执行指令来到达这些路径点）。
    -   **数据处理管道**: 一个组件负责从数据源加载数据并清洗（管理数据状态），另一个组件负责对清洗后的数据进行统计分析（执行业务逻辑）。

## 方案C（自动触发+外部触发双模式）的可复用思想

**来源**: `integrated_system.py` 方案C实现
**日期**: 2026-02-27

### 5. 双模式触发：自动+手动 (Dual-Mode Triggering)

系统在状态机的关键转换点自动触发功能，同时保留外部手动触发的API。这种设计在嵌入式和IoT系统中非常常见。

-   **核心思想**: 对于一个系统内部完全有能力自主判断触发时机的功能，不应将触发决策权完全交给外部。最佳实践是**自动为主、手动为辅**——默认情况下系统自主运行，但保留外部干预的接口，用于特殊场景或人工修正。
-   **实现要点**:
    -   **标志位防重复**: 使用 `_body_shape_auto_triggered` 标志位，确保同一次入座只自动触发一次。
    -   **生命周期重置**: 在所有"离开"转换点（进入 RESETTING/OFF_SEAT）重置标志位，使下次入座可以重新触发。
    -   **配置开关**: 通过 `auto_trigger: true/false` 配置项控制，保持向后兼容。
-   **可复用场景**:
    -   **自动驾驶**: 系统自动检测到前方障碍物时自动刹车，但驾驶员仍可随时踩刹车手动干预。
    -   **工业控制**: 设备在温度超标时自动降功率，但操作员也可以手动触发降功率。
    -   **智能灯光**: 根据光线传感器自动开关灯，但用户也可以通过开关手动控制。

### 6. 精简输出与向后兼容 (Lean Output with Backward Compatibility)

将大量内部状态精简为三个语义清晰的独立字段对外暴露，同时保留所有旧字段确保向后兼容。

-   **核心思想**: 对外接口应遵循**最小知识原则 (Principle of Least Knowledge)**——调用方只需要知道它需要的信息。但在演进过程中，不能破坏已有的集成方。因此采用"新增不删除"的策略：新增精简字段，保留旧字段。
-   **实现要点**:
    -   **语义分组**: 按照使用场景将输出分为独立的语义组（离座状态、体型信息、气囊指令），每个组是一个自包含的字典。
    -   **布尔快捷方式**: 在 `seat_status` 中同时提供枚举值 `state` 和布尔快捷方式 `is_off_seat`/`is_seated`/`is_resetting`，降低调用方的判断复杂度。
    -   **渐进式迁移**: 旧字段标记为"兼容字段"，新集成方使用新字段，旧集成方无需修改。
-   **可复用场景**:
    -   **API版本管理**: REST API的v1和v2共存，v2新增精简响应格式，v1保持不变。
    -   **传感器数据协议**: 新版协议增加聚合字段（如均值、状态标志），旧版原始数据字段保留。
    -   **日志系统**: 新增结构化日志字段，同时保留原始文本日志格式。

### 7. 最小集成包思维 (Minimal Integration Package)

将一个复杂系统的运行时依赖提取为最小文件集，使集成方无需关心开发时的辅助文件。

-   **核心思想**: 区分**开发时依赖**和**运行时依赖**。集成方只需要运行时依赖。通过分析 import 链，递归收集所有必要的 `.py` 文件、模型文件和配置文件，排除测试、可视化、服务端等非必要文件。
-   **实现要点**:
    -   从入口文件（`integrated_system.py`）出发，递归分析 `import`/`from ... import` 语句。
    -   区分静态导入和条件导入（如 `tap_massage.py` 是条件导入，但仍需包含）。
    -   包含模型文件（`.pkl`）和配置文件（`.yaml`）。
    -   生成精简的 `requirements.txt`，只包含运行时依赖。
-   **可复用场景**:
    -   **SDK发布**: 从一个大型项目中提取SDK，只包含公共API和必要依赖。
    -   **模型部署**: 将训练代码和推理代码分离，部署包只包含推理所需的模型和预处理代码。
    -   **嵌入式交叉编译**: 只将目标平台需要的库和资源打包到固件中。

## 7. 状态机离场清理模式（State Machine Exit Cleanup）

**来源**：离座/复位时体型三分类结果未重置的 bug 修复

**问题**：状态机中某些子系统的状态在"离场"（如离座、断开、退出）时没有被正确清理，导致下一次"入场"时残留了上一次的状态数据。

**可复用模式**：

1. **离场清理清单**：每个状态机的"退出转换"都应有一个明确的清理清单，列出所有需要重置的子系统。这个清单应该在设计阶段就确定，而不是在发现 bug 后补充。

2. **API 的 None 安全性**：任何 `set_xxx()` 方法都应考虑 `None` 作为合法输入（表示"清除"），而不是只接受有效值。类型签名应为 `Optional[T]` 而非 `T`。这是一个常见的接口设计盲点——开发时只考虑了"设置"场景，忽略了"清除"场景。

3. **对称性检查法**：review 状态机修改时，对每一个"设置标志"的路径，都检查是否存在对应的"重置标志"路径。画出状态转换图，标注每个转换点的设置/重置操作，确保对称。

**适用场景**：所有包含状态机和子系统联动的嵌入式/实时系统。

## 8. 竞争路径分析法（Competing Path Analysis）

**来源**：调试离座路径时发现的 ADAPTIVE_LOCKED → CUSHION_ONLY → RESETTING 竞争问题

**问题**：当多个条件同时满足时（如坐垫和靠背同时为0），状态机中多个独立的 `if` 分支可能产生竞争，实际走的路径取决于哪个计数器先达到阈值。

**可复用模式**：

1. **独立 if vs elif 的区别**：同一个状态下的多个条件检查，如果用独立 `if` 而非 `elif`，则可能在同一帧内触发多个转换。必须明确这是有意设计还是潜在 bug。

2. **阈值竞争分析**：当两个计数器同时递增时，先到阈值的那个决定实际路径。必须分析所有阈值组合，确认每条可能的路径都有正确的清理逻辑。

3. **中间状态的清理覆盖**：如果 A→B→C 是一条可能的路径，那么 B 状态的转换点也需要有完整的清理逻辑，不能只在 A→C 的直接路径上做清理。

**适用场景**：多条件并行检测的状态机、多传感器融合系统。

## 9. 先验信息约束观测（Prior-Constrained Observation）

**来源**：鲁棒品味记录功能（V1.1）
**日期**：2026-03-05

**问题**：品味记录时用户可能乱动，导致采集到的压力比例值偏离真实舒适状态，最终生成错误的品味区间。

**核心思想**：利用已知的先验信息（用户的充放气操作次数 + 当前自适应阈值），推算出采集数据的合理范围（置信区间），对超出范围的观测值进行抑制或修正。这是贝叶斯推断思想在工程中的简化应用。

**可复用模式**：

1. **乘法因子模型**：当被观测量是一个比例（ratio）时，操作对其的影响是乘法关系而非加法关系。例如充气1次使比例变为原来的 `1.05` 倍，而不是增加 `0.05`。这在处理比例、百分比、倍率等非线性量时是一个常见的建模技巧。

2. **截断 vs 卡尔曼融合**：两种处理异常观测的策略各有适用场景：
   - **截断（Clamp）**：简单粗暴，将超出范围的值强行修正为边界值。适用于异常可能连续出现的场景（如用户持续乱动），因为它不会累积漂移。
   - **卡尔曼融合（Kalman）**：用预测值和观测值做加权融合，观测越偏离预测权重越低。适用于偶发异常的场景，能保留更多有效信息。但在连续异常时方差会膨胀导致漂移。

3. **置信区间的构建范式**：`预期中心 = 基线 × (1 + step)^n`，`区间 = [中心 × (1-tol), 中心 × (1+tol)]`。这个范式可以推广到任何"已知操作次数，需要约束观测范围"的场景。

**适用场景**：
- **传感器标定**：已知施加了N次标准激励，采集到的响应应落在理论值附近，超出范围的视为异常。
- **工业质检**：已知加工了N道工序，产品尺寸应在公差范围内，超出的标记为不良品。
- **金融交易**：已知用户近期的交易模式，突然出现大幅偏离的交易标记为可疑。
- **机器人定位**：已知执行了N步运动指令，里程计读数应在预期范围内，超出的用IMU数据修正。

## 10. 配置驱动的算法切换（Configuration-Driven Algorithm Switching）

**来源**：鲁棒品味记录中截断/卡尔曼两种模式的配置化
**日期**：2026-03-05

**核心思想**：当一个功能有多种算法实现时，不要硬编码选择哪一种，而是通过配置文件来切换。这样可以在不修改代码的情况下，根据实际场景选择最合适的算法。

**实现要点**：
- 所有算法实现共享相同的输入输出接口
- 配置文件中用一个字符串字段指定当前使用的算法名
- 代码中用 `if/elif` 或字典映射来分发到具体实现
- 算法特有的参数也放在配置文件中，用算法名作为前缀区分

**可复用场景**：
- **滤波器选择**：低通/中值/卡尔曼滤波器通过配置切换
- **分类器选择**：SVM/随机森林/神经网络通过配置切换
- **通信协议**：UART/SPI/I2C通过配置切换
- **日志级别**：DEBUG/INFO/WARNING通过配置切换

## 11. GUI中的有限状态动作控制（Timed Action with Auto-Hold）

**来源**：visualizer.py 手动气囊控制面板
**日期**：2026-03-05

**核心思想**：在GUI中执行一个有时间限制的动作时，使用"发送动作→定时器→自动恢复"的三步模式，同时通过按钮禁用防止重复触发。这是一种常见的**防抖+定时回调**模式。

**实现要点**：
- 用户点击按钮后立即发送动作指令（如充气）
- 同时禁用所有相关按钮，设置`action_in_progress`标志
- 使用`root.after(duration_ms, callback)`注册定时回调
- 回调中发送恢复指令（如保持），解除按钮禁用
- 定时器ID保存以便在窗口关闭时取消

**可复用场景**：
- **电机控制面板**：点击"正转"后电机转1秒自动停止
- **阀门控制**：打开阀门N秒后自动关闭
- **LED测试**：点亮LED指定时间后自动熄灭
- **任何需要"脉冲式"控制的嵌入式调试工具**

## 12. 操作计数器驱动的置信区间（Operation-Count-Driven Confidence Intervals）

**来源**：手动模式气囊操作计数 → 品味记录置信区间
**日期**：2026-03-05

**核心思想**：将用户的离散操作（充气/放气次数）转化为连续的预期值范围，用于后续数据采集时的异常过滤。操作次数越多，预期偏移越大，置信区间也相应调整。这是一种**先验信息约束采集**的具体应用。

**实现要点**：
- 维护一个按区域分类的操作计数字典
- 每次操作自动更新对应区域的计数
- 记录品味时，将计数字典传入，系统据此构建置信区间
- 预期值 = 基线中心 × (1 + step_factor)^(净操作次数 × 方向)
- 置信区间 = [预期值 - tolerance, 预期值 + tolerance]

**可复用场景**：
- **标定流程**：已知执行了N次标定动作，预期传感器读数应在理论值附近
- **累积控制**：已知发送了N次加热指令，温度应在预期范围内
- **用量追踪**：已知注入了N次药剂，浓度应在预期范围内

## 13. 接口契约验证（Interface Contract Verification）

**来源**：Bug2修复 — body_shape_classifier返回值缺少status字段导致品味面板不显示体型
**日期**：2026-03-05

**核心思想**：当模块A依赖模块B的返回值中的某个字段时，必须确保该字段确实存在于B的实际返回值中。仅靠"约定"或"注释"是不够的，应通过以下方式之一来保证接口契约的正确性。

第一种方式是**防御性编程**：不依赖特定字段的存在，而是用更宽泛的条件判断。例如本次修复中，将 `body_shape_result.get('status') == 'COMPLETED'` 改为 `body_shape_result is not None`，因为 `update()` 方法只在分类完成或超时时才返回非None值，这个行为本身就是"完成"的信号。

第二种方式是**集成测试**：编写测试用例验证模块A和模块B的实际交互，确保返回值格式符合预期。本次新增的 `test_bug2_body_shape_activation.py` 就是这种方式。

第三种方式是**类型标注+静态检查**：使用TypedDict或dataclass定义返回值结构，配合mypy等工具在编译期发现不匹配。

**可复用场景**：任何多模块协作的系统中，模块间的数据传递都应遵循此原则。特别是当模块由不同人或不同时期开发时，接口契约的隐式假设极易被打破。

## 14. sklearn 模型的跨平台部署策略（ONNX Export Pattern）
**来源**：体型分类模型从 pkl 导出为 ONNX，支持 Android 部署
**日期**：2026-03-05

**核心思想**：scikit-learn 模型通过 pickle 序列化后，部署时需要完整的 sklearn + scipy 环境，这在嵌入式/移动端（Android Termux、Chaquopy）上非常困难。通过 ONNX 中间表示 + onnxruntime 推理，可以将 ML 推理的运行时依赖降到最低。

**实施步骤**：
1. **管线识别**：确定哪些步骤依赖 sklearn（本例：StandardScaler → SelectKBest → KNN），哪些是纯计算（特征工程）
2. **ONNX 导出**：使用 `skl2onnx.convert_sklearn` 将 sklearn Pipeline 导出为 ONNX，指定输入 shape 和类型
3. **参数文件**：将特征列顺序、模型元信息导出为 JSON，确保推理时的特征对齐
4. **纯实现替代**：将依赖 scipy 的计算（如 `stats.skew`、`stats.kurtosis`）用纯 numpy 重写
5. **双后端设计**：优先 ONNX，fallback 到 pkl，保证开发环境和部署环境都能工作
6. **一致性验证**：对比两个后端的输出，确保概率差异在浮点精度范围内

**关键注意点**：
- KNN 模型的 ONNX 导出需要 `zipmap=False` 选项，否则概率输出格式不一致
- 特征列顺序必须与训练时完全一致，通过 `model_params.json` 固化
- 偏度/峰度的纯 numpy 实现需要使用 `ddof=0`（总体标准差），与 scipy 默认行为一致

**效果**：
- 模型文件：43.2 KB (pkl) → 19.9 KB (onnx)
- 运行时依赖：sklearn + scipy + pandas → numpy + onnxruntime
- 推理一致性：概率差异 < 0.003

- 任何需要将 sklearn 模型部署到资源受限环境的场景
